# RaioPDF Architecture

RaioPDF is a fully local desktop PDF suite. The product is organized around three tiers: a Tauri desktop shell, a bundled Stirling-PDF sidecar, and a bundled OCR toolchain. The shell owns windows, menus, file access, and the React UI. The Stirling sidecar runs on localhost inside the installed app and provides the heavier PDF operations that should stay outside the UI process. The OCR toolchain is bundled alongside the app so searchable-PDF workflows can run without cloud services, accounts, telemetry, or user-managed system dependencies.

The TypeScript workspace is organized around a `PdfEngine` seam in `packages/engine-api`. The UI depends on that interface, not on any particular engine implementation. `packages/engine-local` is the in-process implementation using `pdf-lib` (fast structural operations: merge, split, stamps, binders); `packages/engine-sidecar` is the HTTP client for the bundled Stirling-PDF backend, keeping document bytes client-side behind the same opaque handle pattern. That keeps UI workflows stable regardless of which engine serves an operation.

Both implementations are live. The desktop shell launches the bundled Stirling engine on demand behind an authenticated loopback proxy — the UI's engine bridge receives a port plus a per-boot token from the shell and routes the heavier operations (OCR, redaction rasterization, PDF/A conversion, compression, decryption, sanitizing, repair) through the sidecar. The shared launch/supervise/proxy logic lives in the Rust `engine-sidecar-core` crate, used by both the shell and `apps/engine-host` — the standalone host that gives the MCP connector its own engine when the app isn't running (see [`MCP.md`](MCP.md)).

The UI must never call the Stirling sidecar directly. All document operations go through `PdfEngine` so handles, byte serialization, errors, and future sidecar behavior stay behind one boundary. This also keeps the Tauri shell free to decide how local services are launched, discovered, supervised, and shut down without leaking those mechanics into React components.

Desktop packaging has an equally strict platform boundary. The neutral Tauri
configuration cannot bundle by itself; Windows x64 and macOS arm64 builds apply
separate overlays that select exactly one namespaced payload and remap it to the
installed `payload/` resource path. Assemblers, pins, caches, staging paths,
updater keys, and public artifact names derive from `installer/platforms.mjs`.
RaioPDF does not produce a universal bundle or combine Windows and Mac runtimes
in one download.

Large-document path ops are a separate shell-owned lane. The renderer passes opaque file grants to Tauri commands; Rust resolves grants, runs file-to-file work in private temp directories, verifies outputs, and returns fresh output grants that reopen through the same Save As funnel as normal generated documents. Most path ops use qpdf, Ghostscript, or OCRmyPDF directly. PDF Security also lives in this lane: the Windows shell binds an opaque, one-use save target to snapshots of the source files, sends passwords to qpdf only through stdin, creates an AES-256 candidate, verifies its encryption and permissions, rechecks the sources and destination, and only then publishes the protected copy. Password-bearing inspect, decrypt, and protect requests that cross the authenticated loopback proxy use a bounded binary envelope rather than URLs, headers, logs, or JSON. Combine with Exhibits and streamed in-app annotation saves use the existing MCP one-shot launcher as a Node lane: the shell writes below-threshold exhibit/edit payloads to temp files, passes above-threshold exhibit grants as resolved input paths, passes the streamed main-document path to `--one-shot build_binder` or `--one-shot apply_edits`, sets the Node heap/security policy, enforces the aggregate shell-owned size ceiling before pdf-lib reads the files, verifies the output with qpdf, and only then grants the output.

Stirling-PDF code may be consumed only from MIT-licensed directories. Do not copy, vendor, adapt, or link code from `frontend/editor/src/desktop`, `app/proprietary`, `app/saas`, `engine/`, or any other carved-out or non-MIT area. If a Stirling source path is not clearly covered by the MIT license grant, treat it as unavailable for RaioPDF.
